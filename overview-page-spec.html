1 · Overview

The SRE Overview Dashboard is the default landing page of the Environment Management System. It gives an at-a-glance picture of the entire environment estate — health, topology, discovery pipeline status, scheduled maintenance, governance hygiene, and a unified activity feed.

Unlike other pages that make live REST calls to Lambda, this page is snapshot-driven: a background Lambda (overview-snapshot) runs every 15–30 minutes, collects data from all sources, serialises it as a single JSON document, uploads it to S3, and records a pre-signed URL in AWS AppSync (GraphQL API). The React component fetches that URL and renders the snapshot. A WebSocket subscription notifies the page when a fresh snapshot is available.

Key principle: All data for this page comes from a single snapshot JSON object retrieved via a two-stage fetch (AppSync → S3). There are no direct Neptune / Lambda calls during normal rendering.

2 · Architecture — Snapshot Pattern

2.1 Two-stage fetch flow

1 · GET AppSync
getLatestSnapshot
→
2 · S3 presigned URL
snapshot JSON
→
3 · Render all
sections from snapshot

Step 1 is a GraphQL query to AppSync that returns metadata: a snapshot ID, generation timestamp, and an S3 presigned URL valid for 7 days (604,800 s). Step 2 fetches the raw JSON from S3 using that URL (no AWS auth required). Step 3 is pure client-side rendering with no further network calls.

2.2 Real-time subscription

After the initial render, the client opens an AppSync WebSocket subscription (graphql-ws sub-protocol). When the backend generates a fresh snapshot, AppSync pushes a onSnapshotCreated event containing the new presigned URL. The client re-fetches and re-renders automatically — effectively giving a 15–30 min auto-refresh without polling.

The subscription is idempotent: calling the setup function multiple times (e.g., on tab switch) must not open a second WebSocket. Guard with a module-level reference.

2.3 Configuration constants

Constant Value / CDK Output Notes
OV_APPSYNC_URL CDK output: SRE-POC-DashboardApiUrl Full GraphQL endpoint URL ending in /graphql
OV_APPSYNC_KEY CDK output: SRE-POC-DashboardApiKey Sent as x-api-key header; rotate in AWS Console if expired

On startup, check that both constants are set and do not contain the placeholder string "REPLACE_WITH". If not configured, display a setup notice in the header subtitle and skip loading.

3 · AppSync & S3 API Calls

3.1 GetLatestSnapshot — GraphQL Query

GraphQL POST {OV_APPSYNC_URL} Headers: x-api-key + Content-Type: application/json

Request body

"query": "query GetLatestSnapshot {
  getLatestSnapshot {
    id
    generatedAt
    presignedUrl
  }
}"

Success response — HTTP 200

{
  "data": {
    "getLatestSnapshot": {
      "id":           "snapshot-20260527T143000Z",
      "generatedAt": "2026-05-27T14:30:00.000Z",    // ISO timestamp
      "presignedUrl": "https://s3.amazonaws.com/…?…"  // short-lived URL
    }
  }
}

If getLatestSnapshot returns null, no collection run has completed yet. Show: "No snapshot available yet — waiting for first collection run."

Error cases

HTTP status Cause User message
401 / 403 API key invalid or expired "AppSync API key is invalid — rotate the key in AWS Console and update OV_APPSYNC_KEY"
200 + errors[] GraphQL resolver error Show errors[0].message
Any other non-200 Network / gateway issue "AppSync HTTP {status} — {body}"

3.2 Fetch Snapshot JSON from S3

GET {presignedUrl} No auth headers — URL self-authorises

Standard fetch(presignedUrl) — no custom headers required. The URL is generated by the Lambda with GetObjectCommand and is valid for 7 days (604,800 s).

On non-200 response: throw "Failed to fetch snapshot from S3: {status}" and display in the header subtitle. The full snapshot JSON schema is described in §4.

3.3 Real-time WebSocket Subscription

WebSocket wss://{host-id}.appsync-realtime-api.{region}.amazonaws.com/graphql

The realtime URL is derived from the HTTP URL by replacing https:// with wss:// and appsync-api with appsync-realtime-api. Sub-protocol: graphql-ws. Connection payload must include: 

// Derive realtime URL from HTTP AppSync URL:
const rtUrl = OV_APPSYNC_URL
  .replace("https://", "wss://")
  .replace("appsync-api", "appsync-realtime-api");

// Connection initiation payload (base64-encoded in sub-protocol header):
{
  "host": "<appsync-hostname>",
  "x-api-key": OV_APPSYNC_KEY
}

// Subscription message:
{
  "id": "sub-1",
  "type": "subscribe",
  "payload": {
    "data": "{\"query\":\"subscription { onSnapshotCreated { id generatedAt presignedUrl } }\"}",
    "extensions": {
      "authorization": {
        "x-api-key": OV_APPSYNC_KEY,
        "host": "<appsync-hostname>"
      }
    }
  }
}

When a data message is received on the subscription, extract payload.data.onSnapshotCreated.presignedUrl, fetch from S3, and call the full render function again to refresh the dashboard.

4 · Snapshot JSON Schema

The snapshot written to S3 is a slim JSON object with only two top-level keys. Raw collector arrays are used internally by the Lambda to populate computed and are never written to S3. 

{
  "generatedAt":  "2026-06-05T14:30:00.000Z",  // ISO timestamp set by the Lambda
  "computed": { /* §4.13 — all display data, see below */ }
}
Raw data keys removed from payload. envData, networkData, historyData, reconcilerData, and all other raw collector arrays are not present in the S3 object. The frontend must read exclusively from snapshot.computed.*. The sections below (§4.1–§4.12) document the internal shapes for Lambda developer reference only.

4.1 envData — Environment list

[
  {
    "id":    "env-abc123",
    "name":  "Production EU",
    "owner": "platform-team",     // optional; used by hygiene check
    "status":"Active"              // Active | Degraded | Critical | Inactive
  }
]

4.2 integrationsData — Integration list

[
  {
    "id":   "int-xyz789",
    "name": "PaymentGateway→OrderService"
  }
]

4.3 healthData — Health monitoring entities

{
  "totalWithStats": 42,
  "totalEntities":  50,
  "entities": [
    {
      "id":   "env-abc123",
      "name": "Production EU",
      "type": "Environment",          // "Environment" | "Application"
      "stats": {
        "cpu":             "72.4",       // string; percentage
        "memory":          "58.1",
        "latency":         "145",        // milliseconds
        "availability":    "99.8",       // percentage
        "healthScore":     "88",         // 0–100
        "lastSyncedAt":    "2026-05-27T14:28:00Z",
        "collectionTime":  "2026-05-27T14:28:00Z"
      }
    }
  ]
}

Alert derivation rules (computed client-side from stats):

Stat Threshold Severity label
cpu > 80% Critical
memory > 90% Critical
latency > 200 ms Warning
availability < 99.5% Warning
healthScore < 50 Critical
The "Active Alerts" table is entirely client-derived from the health entities — no separate alerts API exists. Multiple stats on the same entity may each produce a separate alert row.

4.4 networkData — Topology graph data

{
  "nodes": [
    {
      "id":           "env-abc123",
      "label":        "Production EU",
      "type":         "Environment",   // "Environment" | "Integration" | "Application"
      "status":       "active",         // lowercase: active | degraded | critical | inactive
      "currentState": "warning",        // performance state: "critical" | "warning" | ""
      "properties": { "status": "Active" }
    }
  ],
  "edges": [
    { "source": "env-abc123", "target": "env-def456" }
  ]
}
The Live Topology Graph renders only critical / warning nodes and their immediate neighbours. If no such nodes exist, the graph shows a green "All environments are healthy" message — it does not fall back to showing all nodes.

4.5 historyData & integrationHistoryData — Change events

[
  {
    "timestamp":   "2026-05-27T12:00:00Z",
    "action":      "create-integration",   // see action vocabulary below
    "entityId":    "env-abc123",
    "entityName":  "Production EU",
    "user":        "alice@example.com",     // optional
    "changes": {
      "integrationName": "PaymentGateway",
      "targetEnvName":   "Staging EU",
      "from":            "OldName",  // for rename
      "to":              "NewName"
    }
  }
]

Action vocabulary

Action string Colour Description
create green Environment created
delete red Environment deleted
rename purple Environment renamed; changes.from / changes.to
update / update-config amber Config property updated
create-integration / add-integration green Integration link added
delete-integration / remove-integration red Integration link removed
update-integration amber Integration properties updated
(any other) blue Generic change event

integrationHistoryData contains the same shape but filtered to integration events only. When present it is used for the Integration Changes chart; otherwise historyData is used as a fallback.

4.6 pipelineData — Discovery pipeline runs

[
  {
    "sourceType": "DCM",               // snake_case → formatted by client
    "totalRuns":  12,                  // total runs ever recorded for this source
    "latestRun": {
      "pipelineId":          "DCM#env-abc123",   // format: SOURCETYPE#envId
      "runId":               "2026-05-27T14:00:00Z", // ISO string = run timestamp
      "status":              "completed",       // completed | failed | running
      "triggeredBy":         "scheduled",
      "envsCreated":         2,
      "integrationsCreated": 5,
      "integrationsStale":   0,
      "summaryErrors":       0,
      "durationMs":          4820,              // optional; milliseconds or null
      "error":               null
    },
    "recentRuns": [ /* last 5 runs (newest first), same shape as latestRun */ ]
  }
]

4.7 scheduleData — Maintenance windows

[
  {
    "envId":         "env-abc123",
    "envName":       "Production EU",
    "owner":         "platform-team",          // optional
    "collaborators": "alice@example.com",       // optional; raw stored string
    "schedule":      "[{\"date\":\"2026-06-01\",\"startTime\":\"22:00\",\"endTime\":\"00:00\",\"description\":\"Patch\"}]",  // raw stored value — see formats below
    "windowStart":   "2026-06-01T22:00:00Z",    // parsed from schedule by Lambda
    "windowEnd":     "2026-06-02T00:00:00Z",    // parsed from schedule by Lambda
    "description":   "Quarterly patch window",   // parsed from schedule by Lambda
    "conflict":      false,                      // true when overlap detected (pre-computed)
    "conflictWith":  ""                          // name of conflicting env, "" if none
  }
]
schedule field formats: The Lambda parses two raw input formats into windowStart / windowEnd / description. The raw string is preserved in schedule for reference. Supported formats:
  • JSON array: [{"date":"2026-06-01","startTime":"22:00","endTime":"00:00","description":"..."}] — supports multiple windows per environment
  • Text: "Apr 12 02:00-04:00 UTC — description text"
Each window in a JSON array produces a separate item in scheduleData.

4.8 degreeData — Blast radius / connection counts

{
  "env-abc123": 18,   // entity ID → total edge count
  "env-def456": 12
}

Top 8 entries by degree are rendered. Entities with ≥ 10 connections are flagged as potential Single Points of Failure (SPOF).

6 · Functional Requirements

6.1 Header Bar

  • Title: "SRE Overview Dashboard"
  • Sub-title (id: ov-last-updated): dynamic text — "Last synced N min ago · auto-refreshes every 15–30 min". Shows "Loading snapshot…" during fetch.
  • Live badge — always visible once configured.
  • ↻ Refresh button — calls loadOverviewDashboard(); re-fetches the latest snapshot from AppSync.

6.5 Live Topology Graph — D3 Force-directed

Library: D3.js v7. SVG element id: ov-network-svg, height: 260px.

Source: snapshot.computed.topologyFocus. The Lambda has already filtered to alert nodes + their one-hop neighbours. No client-side node selection logic is needed.

Rendering

  • If topologyFocus.allHealthy === true: skip the simulation entirely and render the green message in the SVG canvas: "✓ No critical or warning nodes · All environments are healthy"
  • Otherwise: feed topologyFocus.nodes and topologyFocus.edges directly to D3 — no further filtering
  • Badge text (ov-graph-badge): render topologyFocus.badgeText directly

Visual encoding

Node type Radius Fill Opacity Extra ring
Environment (_isCritical) 11 #dc2626 1.0 Dashed ring r+5, 35% opacity
Environment (_isWarning) 11 #d97706 1.0 Dashed ring r+5, 35% opacity
Environment (neighbour — !_isAlert) 11 #2563eb 0.55 None
Integration / Application 7 #0891b2 / #7c3aed varies None

Edge colours

  • Edge to/from a critical node → stroke #fca5a5, dashed array off
  • Edge to/from a warning node → stroke #fde68a
  • Neighbour-to-neighbour → stroke #cbd5e1, dash-array 4 2

Force simulation parameters

  • Link distance: 72, strength: 0.65
  • Charge: −260
  • Collision radius: node.r + 8

Supports zoom (scale 0.4–2.5) and node drag. Tooltip on hover shows name + state.

Badge text (id: ov-graph-badge): "{criticalCount} critical · {warnCount} warning · {neighbourCount} connected"
